Overview

This plan turns the whole-repository code review into a sequenced, milestone-based work list. It covers all 17 findings (1 High, 6 Medium, 7 Low, 3 Info). The ordering is by risk, then theme: correctness and stability fixes come first because they are safe, self-contained, and high-value; the low-risk energy wins follow; then performance polish; then the larger architectural energy redesign as its own phase; and finally housekeeping and documentation. Each finding maps to exactly one milestone so it can be reviewed and committed on its own.

Every milestone is small enough to land as a single commit and carries its own live status, implementation notes, and review-comments fields, updated in place as work proceeds. Where a change touches an always-on 1 Hz hot path, the plan asks for a powermetrics/Instruments before-and-after measurement so the energy effect is verified, not assumed.

Questions & answers

Which findings should the plan cover?

All 17 findings, including the three Info items (#15 deprecated constant, #16 per-tick heap churn, #17 no teardown of detached threads/singletons).

How far should the plan go on finding #5 (three unsynchronized ~1 Hz wake loops)?

Do the low-risk per-instrument cadence changes now as an early energy milestone, and keep the single-timer coalescing plus event-driven notifications (IOPSNotificationCreateRunLoopSource, SCNetworkReachability) as a clearly-separated later phase.

How should milestones be ordered / grouped?

By risk, then theme: correctness/UB/stability first, then energy wins, then performance polish, then the energy redesign, then Info/housekeeping — grouped into named phases.

Notes

  • Do not undo prior energy work. The interval bump to 1 s (d413715), the SMC key-read reduction (209e318), and the sleep/wake gating (c9ec26d) are the right direction; these milestones build on them.
  • Runtime model. main starts the enabled observers, then Screen::start(1000) drives an ncurses render loop at ~1 Hz. Observers run on two other threads — a dedicated CPULoad sampling thread and the shared UpdateQueue thread (GPU, memory, battery, temperature, network). A fourth thread runs the SleepManager run loop.
  • Verification. After each milestone: build the statusbar scheme with xcodebuild, run the widget, and confirm the affected display still renders correctly. For energy milestones (#2, #3, #5, #8), capture a powermetrics/Instruments sample before and after and record the delta in the milestone's implementation notes.
  • Commit granularity. One milestone = one commit. The user commits; do not commit or push automatically.
  • Coding conventions. Follow the existing house style (PIMPL, one type per file, early returns, aligned assignments, no throwing from public methods, prefer const). Keep changes localized to the finding at hand — no unrelated refactors.
  • Convergence between phases. Milestone 10 (cadence) and Milestone 13 (blocking poll()) both touch the loop timing that Phase 4 will restructure. Land them first; Phase 4 then subsumes them into a single coalesced timer.

M2

Fix the mach_host_self() port-right leak

Medium
statusbar/SB/Instruments/CPULoad.cpp:251 statusbar/SB/Instruments/MemoryInfo.cpp:244 — review #4

Goal

Stop leaking a host-port send right twice per second. Unlike mach_task_self(), the right returned by mach_host_self() is owned by the caller and must be balanced.

Tasks

  • Cache the host port once per file (static const mach_port_t host = mach_host_self();) or deallocate after each use with mach_port_deallocate( mach_task_self(), host ).
  • Apply the same fix in both getCPULoadInfo() and getMemoryInfo(); keep the approach identical in both for consistency.
  • Confirm no other call site takes an unbalanced mach_host_self() right.
  • Sanity-check port count over time (e.g. observe Mach port count does not grow while running).

Completed

Implementation notes

Chose the cache-once option: added a function-local static const mach_port_t host = mach_host_self(); to both getCPULoadInfo() (CPULoad.cpp:242) and getMemoryInfo() (MemoryInfo.cpp:241), and replaced the inline mach_host_self() arguments to host_processor_info() and host_statistics() with host. This acquires exactly one host-port send right per process for its lifetime instead of a fresh unbalanced right on every poll, eliminating the accumulation. The right is intentionally never deallocated — a single, bounded, process-lifetime hold reclaimed by the kernel at exit (consistent with the widget's run-until-quit model).

The same approach is used in both files for consistency. The two statics are independent (one per function), and C++11 guarantees thread-safe "magic static" initialization, so the different threads that call these getters (dedicated CPULoad thread vs. shared UpdateQueue thread) are safe even on a simultaneous first call. The host name port is stable for the process lifetime, so caching produces identical results to re-fetching. A tree-wide grep confirmed these are the only two mach_host_self() call sites — no other leak remains. The vm_deallocate element-size issue at the end of getCPULoadInfo() is a separate finding handled in M4 and was left untouched.

Re-aligned the MemoryInfo count/vmStat declaration group so the new line keeps the = aligned (house style). Verified with xcodebuild ... -configuration Debug build: BUILD SUCCEEDED with no warnings under -Werror.

Automated review loop: 1 round, converged. The fresh-context reviewer confirmed the leak fix, thread-safety, unchanged behavior, cross-file consistency, absence of other call sites, and no regressions/style issues. No Critical/High/Medium/Low/Info findings.

Phase 4 — Energy architecture redesign

The larger, higher-risk restructuring deferred from finding #5: coalesce the independent ~1 Hz wake loops into a single timer and move slowly-changing sources onto event notifications. Land only after Phases 1–3 and their measurements are in.

M15

Coalesce wakeups into a single timer ENERGY

Medium
statusbar/SB/UI/Screen.cpp:191–257 statusbar/SB/UpdateQueue.cpp:107–140 statusbar/SB/Instruments/CPULoad.cpp:192–238 — review #5 (redesign portion)

Goal

Stop the three unsynchronized fixed-interval sleep_for timers (render loop, UpdateQueue, dedicated CPULoad thread) from fighting CPU idle residency. Drive sampling and rendering from a single coalesced timer so wakeups align.

Tasks

  • Design a single scheduler/timer that fires at a base tick and dispatches each instrument on its own multiple of that tick (subsuming the per-instrument cadence from M10).
  • Decide the fate of the dedicated CPULoad sampling thread — fold it into the shared schedule if its two-sample-with-sleep model allows, or document why it must stay separate.
  • Fold the render-loop timing (post-M13) into the same schedule so the UI redraw coalesces with sampling.
  • Preserve sleep/wake gating and clean stop behaviour across the new model.
  • Measure wakeups/idle residency and energy before and after with powermetrics/Instruments; record the delta. This is the milestone whose benefit most needs proof.
