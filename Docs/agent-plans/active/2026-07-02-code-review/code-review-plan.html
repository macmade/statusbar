Overview

This plan turns the whole-repository code review into a sequenced, milestone-based work list. It covers all 17 findings (1 High, 6 Medium, 7 Low, 3 Info). The ordering is by risk, then theme: correctness and stability fixes come first because they are safe, self-contained, and high-value; the low-risk energy wins follow; then performance polish; then the larger architectural energy redesign as its own phase; and finally housekeeping and documentation. Each finding maps to exactly one milestone so it can be reviewed and committed on its own.

Every milestone is small enough to land as a single commit and carries its own live status, implementation notes, and review-comments fields, updated in place as work proceeds. Where a change touches an always-on 1 Hz hot path, the plan asks for a powermetrics/Instruments before-and-after measurement so the energy effect is verified, not assumed.

Questions & answers

Which findings should the plan cover?

All 17 findings, including the three Info items (#15 deprecated constant, #16 per-tick heap churn, #17 no teardown of detached threads/singletons).

How far should the plan go on finding #5 (three unsynchronized ~1 Hz wake loops)?

Do the low-risk per-instrument cadence changes now as an early energy milestone, and keep the single-timer coalescing plus event-driven notifications (IOPSNotificationCreateRunLoopSource, SCNetworkReachability) as a clearly-separated later phase.

How should milestones be ordered / grouped?

By risk, then theme: correctness/UB/stability first, then energy wins, then performance polish, then the energy redesign, then Info/housekeeping — grouped into named phases.

Notes

  • Do not undo prior energy work. The interval bump to 1 s (d413715), the SMC key-read reduction (209e318), and the sleep/wake gating (c9ec26d) are the right direction; these milestones build on them.
  • Runtime model. main starts the enabled observers, then Screen::start(1000) drives an ncurses render loop at ~1 Hz. Observers run on two other threads — a dedicated CPULoad sampling thread and the shared UpdateQueue thread (GPU, memory, battery, temperature, network). A fourth thread runs the SleepManager run loop.
  • Verification. After each milestone: build the statusbar scheme with xcodebuild, run the widget, and confirm the affected display still renders correctly. For energy milestones (#2, #3, #5, #8), capture a powermetrics/Instruments sample before and after and record the delta in the milestone's implementation notes.
  • Commit granularity. One milestone = one commit. The user commits; do not commit or push automatically.
  • Coding conventions. Follow the existing house style (PIMPL, one type per file, early returns, aligned assignments, no throwing from public methods, prefer const). Keep changes localized to the finding at hand — no unrelated refactors.
  • Convergence between phases. Milestone 10 (cadence) and Milestone 13 (blocking poll()) both touch the loop timing that Phase 4 will restructure. Land them first; Phase 4 then subsumes them into a single coalesced timer.

IMPL::run() now publishes _rl and drains _pendingSources (adding each to the loop) inside the same locked block, so a source is never both queued and directly added, and never lost.
  • Instruments simplified: BatteryInfo/NetworkInfo no longer spawn threads or run their own CFRunLoop. On successful source creation they call SB::SleepManager::shared().addRunLoopSource( source ); the <thread> includes were dropped and #include "SB/SleepManager.hpp" added. The created source (and, for network, the SCDynamicStoreRef which the source retains) remain process-lifetime and are intentionally never released. addRunLoopSource is called outside the instrument's own lock, so there is no nested cross-object lock / deadlock.
  • Sleep-gating note now resolved by construction: since the only run loop hosting these sources is SleepManager's own thread, and macOS suspends that thread during real sleep, the earlier "bypasses the sleep gate" Info point no longer applies in practice.

Review of the rework — 2 further rounds. Round A (fresh context) confirmed the locking, CF ownership, header hygiene, -Werror cleanliness and no deadlock, but flagged one Medium: my first cut of run() also ran the loop when IORegisterForSystemPower failed (rootPort == MACH_PORT_NULL); with no power source the loop could be empty, CFRunLoopRun() returns immediately, the thread exits, and _rl would be left dangling on a dead thread — a use-after-free if a source were added later. Fixed by restoring the early return on rootPort == MACH_PORT_NULL before publishing _rl: on that (near-impossible) failure _rl stays null, so addRunLoopSource keeps sources safely queued (unserviced) rather than touching a dead loop — graceful degradation, no UB. The pending-source drain is kept for the normal success path. Round B (fresh context) verified the fix genuinely eliminates the UAF, introduces no new stranding/leak/UB beyond the documented degradation, keeps all _rl/_pendingSources access serialized under _rmtx, and does not regress the sleep/wake lifecycle. Converged. BUILD SUCCEEDED under -Werror throughout.

The two open items above remain: the watch NULL check (Low, unreachable) is unchanged; the sleep-gating Info is now moot per the note above.
