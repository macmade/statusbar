statusbar — Code Review

Generated: 2026-07-02

Scope: Whole repository (all tracked C++ sources under statusbar/).

Primary focus: energy consumption & CPU usage (battery life for an always-on widget), per the requested emphasis.

Summary

statusbar is a small, cleanly-structured C++ ncurses terminal widget that displays CPU, GPU, memory, battery, temperature, network and date/time. The architecture is consistent (PIMPL throughout, one type per file, singleton observers feeding an ncurses render loop) and the author has clearly already invested in energy behaviour: the refresh interval was raised from 500 ms to 1 s, sleep/wake notifications gate the polling loops, and the SMC key set is cached so only temperature keys are re-read. Those are the right instincts and the foundation is solid.

That said, the review surfaced one genuine crash risk (an unchecked ifaddrs dereference on the network path, which is on by default), a handful of latency/undefined-behaviour defects that are masked on typical hardware but real, and — most relevant to your focus — several concrete energy wins. The biggest energy costs are per-key SMC open/close handshakes on every temperature poll, a full IORegistry property copy on every GPU poll, three independent one-second wake loops that never coalesce, and a per-second Mach port-right leak. None of these are architectural dead-ends; each has a localized, low-risk fix.

Notes

  • This is a whole-repository review, not a diff review, so there is no regression section. Where relevant I checked the git history (e.g. d413715 raised the interval to 1 s; 209e318 reduced SMC key reads; c9ec26d added sleep/wake gating) to understand intent, and the findings build on that direction rather than undoing it.
  • Two Mach-API claims below (the mach_host_self() leak and the vm_deallocate size) were verified against external documentation rather than asserted from memory — sources are linked in the Appendix.
  • Runtime model reviewed: main starts the enabled observers, then Screen::start(1000) drives an ncurses render loop at ~1 Hz. Observers run on two other threads — a dedicated CPULoad sampling thread and the shared UpdateQueue thread (GPU, memory, battery, temperature, network). A fourth thread runs the SleepManager run loop.
  • Not covered in depth: the Xcode project/build settings, the xcconfig submodule, and CI configuration — the focus was the application source.

Findings

#1

Unchecked ifa_addr dereference can crash the network poll

High
crashstability statusbar/SB/Instruments/NetworkInfo.cpp:172

Description

getNetworkInfo() walks the getifaddrs() linked list and immediately reads interface->ifa_addr->sa_family. Per the getifaddrs(3) contract, ifa_addr may be NULL for interfaces that have no address assigned (this occurs in practice for certain tunnel/utun and point-to-point interfaces). When such an interface is present, the dereference is a null-pointer read and the process crashes. Because network display is enabled by default (Options turns everything on when no flags are given) and the poll runs once per second, this is a real, recurring crash exposure, not a theoretical one. 

while( interface != nullptr )
{
    if( interface->ifa_addr->sa_family == AF_INET )   // ifa_addr may be NULL
    {
        ...

Recommendation

Guard the pointer before dereferencing, and advance the iterator:

if( interface->ifa_addr != nullptr && interface->ifa_addr->sa_family == AF_INET )
#2

SMC user-client is opened and closed on every key read ENERGY

Medium
performanceenergy statusbar/SB/Instruments/Internal/SMC.cpp:49–58

Description

callSMCFunction() wraps each SMC transaction in a kSMCUserClientOpen → IOConnectCallStructMethod → kSMCUserClientClose sequence — three IOKit user-client round trips per key. readSMCSensors() then reads every cached temperature key once per second. On Apple Silicon there are commonly dozens of T* sensor keys, so steady-state temperature polling costs roughly 3 × N kernel round trips every second (plus a one-time full-keyspace enumeration on the first poll, each of those keys also paying the open/close). For an always-on battery widget this is the single largest avoidable source of wakeups and kernel work.

Recommendation

Hoist the open/close out of the per-key path: the connection is already opened once at init via IOServiceOpen. Do one kSMCUserClientOpen before a read batch and one kSMCUserClientClose after (or open once for the process lifetime), turning 3 × N round trips into N + 2. Combine with #5 (lower temperature cadence) for a compounding win. Measure with powermetrics/Instruments before and after.

#3

GPU poll copies all accelerator properties every second ENERGY

Medium
performanceenergy statusbar/SB/Instruments/GPULoad.cpp:169

Description

getGPULoad() calls IORegistryEntryCreateCFProperties(), which materializes the entire property dictionary of each IOAccelerator entry into CoreFoundation objects, purely to read three numbers out of the nested PerformanceStatistics dictionary. That property blob can be large, and this allocation/serialization happens once per second, per accelerator, for the life of the process.

Recommendation

Fetch only what is needed with IORegistryEntrySearchCFProperty() for the PerformanceStatistics key (with kIORegistryIterateRecursively if required), avoiding the full-dictionary copy. Consider polling the GPU at a lower cadence than 1 Hz as well.

#4

mach_host_self() port-right leak on every CPU and memory poll

Medium
resource leakstability statusbar/SB/Instruments/CPULoad.cpp:251 statusbar/SB/Instruments/MemoryInfo.cpp:244

Description

Both getCPULoadInfo() and getMemoryInfo() call mach_host_self() inline and never release the returned send right. Unlike mach_task_self() — which is a cached global and must not be deallocated — mach_host_self() returns a fresh send right to the host port that the caller owns and must balance with mach_port_deallocate(mach_task_self(), host) (see Appendix for sources). Here it is leaked twice per second. Over the multi-day uptime typical of a status widget this accumulates hundreds of thousands of port references; this exact pattern has caused port-exhaustion bugs in psutil and Chromium.

Recommendation

Cache the host port once (e.g. a file-scope static const mach_port_t host = mach_host_self(); initialized once), or deallocate it after each use: 

mach_port_t host = mach_host_self();
/* ... use host ... */
mach_port_deallocate( mach_task_self(), host );
#5

Three independent ~1 Hz wake loops with fixed sleeps; no coalescing ENERGY

Medium
architectureenergy statusbar/SB/UI/Screen.cpp:191–257 statusbar/SB/UpdateQueue.cpp:107–140 statusbar/SB/Instruments/CPULoad.cpp:192–238

Description

The process wakes at least three times per second on unsynchronized schedules: the render loop (sleep_for(refreshInterval)), the UpdateQueue thread (sleep_for(1s), running GPU/memory/battery/temperature/network back-to-back), and the dedicated CPULoad thread (sample → sleep_for(1s) → sample). Each fixed-interval sleep_for is an independent timer that fights CPU idle residency. Metrics like temperature, battery capacity, memory and network address change far more slowly than 1 Hz, so much of this work is redundant. This is the "many cheap wakeups" pattern that most hurts battery on modern Apple Silicon.

Recommendation

Give each instrument its own cadence instead of a shared 1 s tick — e.g. CPU/GPU at 1–2 s, memory at ~2 s, temperature/battery/network at 5–10 s. Better still, drive everything from a single timer so wakeups coalesce, and for infrequently-changing sources (battery, network) prefer event notifications (IOPSNotificationCreateRunLoopSource, SCNetworkReachability) over polling. Even without a redesign, raising the slow observers' intervals is a low-risk, direct battery win.

#6

String::format() reuses a va_list after it is consumed (UB)

Medium
undefined behaviorstability statusbar/SB/Helpers/String.cpp:40–48

Description

format() calls vsnprintf(nullptr, 0, format, ap) to size the buffer, then calls vsnprintf(cp, size+1, format, ap) with the same ap. Per the C standard, after a function that invokes va_arg (which vsnprintf does internally) returns, the va_list is left indeterminate; using it again without re-initialization is undefined behaviour and can print garbage or crash. This runs every second via bytesToHumanReadable() (memory) and fourCC() (SMC key names, on the temperature path). It happens to work on some ABIs, which is exactly why it is easy to miss. 

int size = vsnprintf( nullptr, 0, format, ap );
...
vsnprintf( cp, size + 1, format, ap );   // ap already consumed → UB

Recommendation

Use va_copy for the sizing pass and keep the original for the real call:

va_list ap, ap2;
va_start( ap, format );
va_copy( ap2, ap );
int size = vsnprintf( nullptr, 0, format, ap2 );
va_end( ap2 );
/* ... allocate ... */
vsnprintf( cp, size + 1, format, ap );
va_end( ap );
#7

vm_deallocate uses the wrong element size (over-deallocates 2×)

Medium
undefined behaviorcorrectness statusbar/SB/Instruments/CPULoad.cpp:276

Description

The buffer returned by host_processor_info() holds cpuLoadInfoCount integer_t (4-byte) elements, so the correct release size is cpuLoadInfoCount × sizeof(integer_t). The code instead passes sizeof(processor_info_array_t) × cpuLoadInfoCount. Since processor_info_array_t is a pointer type (sizeof == 8), this requests deallocation of twice the region that was mapped. On typical core counts the buffer is well under one page and page-rounding hides the error, but on high-core-count machines (many-core Mac Pro / server configurations) the 2× size can extend past the mapped region into an adjacent allocation and unmap memory that was never returned — latent UB. 

vm_deallocate( mach_task_self(),
               reinterpret_cast<vm_address_t>( cpuLoadInfo ),
               sizeof( processor_info_array_t ) * cpuLoadInfoCount );  // 8 * count

Recommendation

vm_deallocate( mach_task_self(),
               reinterpret_cast<vm_address_t>( cpuLoadInfo ),
               cpuLoadInfoCount * sizeof( integer_t ) );
#8

Temperature polled at 1 Hz reads both HID and full SMC sets each tick ENERGY

Low
performanceenergy statusbar/SB/Instruments/TemperatureInfo.cpp:166–199

Description

Each temperature observe reads both the HID sensor set — readHIDSensors() calls IOHIDEventSystemClientCopyServices() and enumerates every matching service, copying a "Product" property and an event per service — and the SMC set (#2), once per second. Temperature is a slowly-varying quantity; sampling it every second doubles the sensor traversal cost for no visible benefit.

Recommendation

Poll temperature on a longer interval (e.g. every 5 s) — this pairs naturally with the cadence change in #5 and multiplies the savings from #2.

#9

Render loop queries all six instruments regardless of enabled options

Low
performance statusbar/main.cpp:96–101

Description

The onUpdate closure unconditionally constructs CPULoad::current(), GPULoad::current(), MemoryInfo::current(), BatteryInfo::current(), NetworkInfo::current() and TemperatureInfo::current() every frame, even when the corresponding option is off. Each current() takes the instrument's recursive mutex and returns a deep copy (PIMPL clone). It is minor, but it is pure waste when only a subset is displayed.

Recommendation

Guard each current() behind its options.xxx() flag so disabled instruments are neither copied nor locked.

#10

ColorPair::pairForColors does a linear scan on every print

Low
performance statusbar/SB/UI/ColorPair.cpp:44–57 statusbar/SB/UI/Window.cpp:96–106

Description

Every colored print linearly scans the 81-entry (9×9) pair vector twice (once for the foreground/background pair, once to reset to clear/clear), doing a Color equality compare each step. A single CPU line issues ~13 prints, and the render runs every second, so this is a few thousand comparisons per second. Additionally Window::print calls Screen::shared().supportsColors() twice per print, each taking a recursive-mutex lock.

Recommendation

Replace the linear search with an O(1) lookup — the pair index is derivable directly from the two color enum values (a 9×9 table), removing both the scan and the repeated hot-path locking.

#11

Non-atomic bool flags read outside the lock (data race)

Low
undefined behaviorconcurrency statusbar/SB/UI/Screen.cpp:191, 249 statusbar/SB/UpdateQueue.cpp:115 statusbar/SB/Instruments/CPULoad.cpp:196

Description

The _sleeping/sleeping and _running flags are written under a mutex (from the sleep-callback thread / stop()) but read without the lock in the loop conditions and the sleep checks. Concurrent unsynchronized access to the same non-atomic object is a data race and therefore undefined behaviour under the C++ memory model, even though it is benign in practice for a single bool on these architectures.

Recommendation

Make these flags std::atomic<bool> (they need no other locking), or read them under the same mutex that guards the write.

#12

Runtime strings passed as printf format strings

Low
stabilitycorrectness statusbar/SB/UI/Window.cpp:82 statusbar/SB/UI/Screen.cpp:151

Description

Window::print(const std::string & s) forwards to ::wprintw(win, s.c_str()) and Screen::print(const std::string & s) to ::printw(s.c_str()) — the string is interpreted as a format string. Today every caller passes literals or values that cannot contain %, so it is latent rather than live, but any future caller passing text with a stray % (a network name, a file path) would read garbage varargs or crash.

Recommendation

Use a fixed format: ::wprintw(win, "%s", s.c_str()) and ::printw("%s", s.c_str()).

#13

Type-punning via reinterpret_cast<float*> in SMC float decode

Low
undefined behavior statusbar/SB/Instruments/Internal/SMC.cpp:273

Description

readFloat32() reinterprets a uint32_t* as float* and dereferences it. Reading an object through a pointer of an incompatible type violates the strict-aliasing rule; it works under clang today but is undefined behaviour and can break under aggressive optimization.

Recommendation

Copy the bits instead: float f; std::memcpy(&f, &u32, sizeof(f)); (or use std::bit_cast in C++20).

#14

Keyboard/resize handled by non-blocking poll then a fixed sleep

Low
responsivenessenergy statusbar/SB/UI/Screen.cpp:216–255

Description

The render loop poll()s stdin with a 0 ms timeout (returns immediately) and then sleep_for(refreshInterval). As a result, a keypress such as q (quit) and terminal resizes are only noticed on the next tick — up to a full second of latency — and the loop wakes on a timer even when nothing is happening.

Recommendation

Block in poll() with the refresh interval as its timeout instead of sleeping separately: the loop then wakes immediately on input and otherwise on the timer, improving responsiveness and removing one redundant wake per cycle.

#16

Per-tick heap churn in every observer

Info
performance CPULoad.cpp:227–235, MemoryInfo.cpp:212–214, GPULoad.cpp:144–146, …

Description

Each observe() does delete + new on the shared info object every cycle, and current() deep-copies it. The allocator traffic is small but avoidable; assigning into an existing object (or storing values directly rather than a heap pointer) would remove it.

Recommendation

Store the info by value and update fields in place under the lock, avoiding the per-tick allocate/free.

#17

Detached threads and singletons are never torn down

Info
architecture UpdateQueue.cpp:139, CPULoad.cpp:80, SleepManager.cpp:166

Description

Observer threads are detach()ed and the singletons are heap-allocated via new in call_once with no matching cleanup. This is a deliberate and acceptable choice for a process that lives until q/kill, but it does mean there is no clean shutdown path and tools like leak checkers will flag the singletons at exit.

Recommendation

No action required for normal use; documenting the intent (or using function-local statics) would make the lifetime model explicit.

Strengths

  • Consistent, disciplined structure. PIMPL everywhere, one type per file, uniform observer pattern, and value semantics with copy/move + swap — easy to read and extend.
  • Energy awareness is already present. Sleep/wake gating (SleepManager), the 500 ms→1 s interval bump, and SMC key caching show the right priorities; the findings above extend that work rather than contradict it.
  • Careful CoreFoundation memory handling. The battery and GPU paths pair CFRelease/IOObjectRelease correctly across early-return branches, and type IDs are validated before casting CF values.
  • Defensive SMC reads. readSMCKey bounds the copy against maxSize and validates result codes, and readSMCKeyInfo caches key metadata to avoid repeat lookups.

Appendix — verification sources

API-behaviour claims that were checked against external documentation rather than asserted from memory:

  • mach_host_self() ownership (finding #4): the returned send right must be released with mach_port_deallocate, unlike mach_task_self(). See psutil #333 (Mach port leak on macOS), mach_override PR #16, and Robert Sesek, "Debugging Mach Ports".
  • host_processor_info deallocation size (finding #7): the region should be freed as count × sizeof(integer_t). See the Mach Kernel Interface Reference and Hammerspoon #1238 (host_processor_info memory handling).
